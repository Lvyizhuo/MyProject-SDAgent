## 1. 背景与结论

### 1.1 背景

当前系统的知识库主要依赖管理员手动上传文档，虽然已经具备文档解析、OCR 兜底、切片、向量化和知识库管理能力，但在业务上存在两个明显问题：

1. 主站的政策查询和政策匹配板块后续都将依赖知识库内容，知识库覆盖面会直接影响产品可用性。
2. 政策来源不会局限于单一网站，后续会逐步扩展到省级厅局网站、地市商务局网站、政策专题栏目、附件公告等多个来源。

因此，知识库不应继续仅作为“管理员上传文档存储区”，而应升级为“多源政策内容供给平台”，统一承担采集、治理、审核、发布和入库职责。

### 1.2 方案选择结论

在此前讨论的两个方向中：

- 方案一：增强型定时采集 + 人工审核入库
- 方案二：事件驱动的内容采集 ETL 平台

对于当前目标，推荐以方案二作为目标架构。

原因如下：

1. 主站已有多个核心业务板块依赖知识库，知识库将从“后台功能”上升为“核心基础能力”，需要稳定性、可追溯性和可治理性。
2. 后续数据源明显会扩展到多站点、多栏目、多附件类型，单纯的定时抓取脚本和人工审核模式很快会成为瓶颈。
3. 方案二天然适合处理多来源、异步任务、失败重试、去重、版本管理和多阶段审核。
4. 后续无论是政策查询还是政策匹配，都需要可解释的数据来源和较稳定的内容时效性，方案二更容易支撑这些要求。

但从实施节奏看，不建议一开始就做成重量级平台，而建议采用：

- 架构目标按方案二设计
- 首期交付按方案一的最小闭环落地

换句话说：

本项目应选择“方案二为主、方案一为首期实现形态”的路线。

## 2. 设计目标

### 2.1 总体目标

建设一个可持续扩展的多源政策知识库采集与治理平台，为以下场景提供统一内容底座：

1. 政策查询
2. 政策匹配
3. 管理员知识库运营
4. 后续的推荐、订阅、政策变更提醒等能力

### 2.2 设计原则

1. 多源统一接入：支持不同政府网站、栏目页、文章页、附件页、静态文档。
2. 原始数据保留：HTML、附件、抽取文本、解析结果全部保留，便于回放和重处理。
3. 入库前治理：先去重、分类、抽取元数据、评分、审核，再进入知识库。
4. 任务异步解耦：采集、解析、OCR、审核、向量化、发布相互解耦。
5. 可观测可追溯：每一条知识内容都能追溯来源、版本、处理日志和审核记录。
6. 业务友好：查询和匹配场景可基于分类、地区、时效、政策类型等结构化字段过滤。

## 3. 为什么方案二更适合

### 3.1 对业务侧更匹配

如果只是补充一个官网专题栏目，方案一足够。但当前明确提出两个业务前提：

1. 政策查询板块要长期依赖知识库。
2. 政策匹配板块也要长期依赖知识库。

这意味着知识库需要承担更高要求：

- 内容覆盖必须持续扩展
- 数据更新必须稳定发生
- 重复和脏数据必须可控
- 来源和发布时间必须可解释
- 内容结构必须足够标准化，便于匹配算法使用

方案一适合解决“有没有”的问题。
方案二更适合解决“能否长期成为基础设施”的问题。

### 3.2 对多源扩展更匹配

未来来源可能包括：

1. 山东省商务厅专题栏目
2. 山东省政府门户政策栏目
3. 山东省发改委、财政厅等厅局栏目
4. 16 地市商务局或政府公开栏目
5. PDF、DOC、DOCX 等附件型政策正文

这些来源的页面结构和更新频率不会一致。

如果继续采用单体定时爬取方案，后续会出现：

1. 规则散落，难以维护
2. 一次抓取失败影响整批任务
3. 难以定位是发现失败、下载失败还是解析失败
4. 难以按来源做审计和回放

方案二通过任务分阶段拆分，更适合长期扩展。

### 3.3 对检索和匹配更匹配

政策查询与政策匹配不只是看“文本是否能检索出来”，还要求：

1. 能区分省级政策、市级政策、解读、活动通知、实施细则
2. 能按地区、时间、品类、适用对象过滤
3. 能知道一条内容是否为最新版本
4. 能控制哪些内容进入主知识库，哪些内容只做备查

这要求知识库前置有内容治理环节，而不是抓完直接 embedding。

## 4. 总体架构设计

### 4.1 分层架构

建议整体拆分为六层：

1. 数据源接入层
2. 采集编排层
3. 内容治理层
4. 审核发布层
5. 知识入库层
6. 检索服务层

### 4.2 各层职责

#### 4.2.1 数据源接入层

职责：

1. 维护站点与栏目配置
2. 按规则发现文章页与附件页
3. 支持不同站点的页面模板适配

建议能力：

1. source_site：站点配置
2. source_section：栏目配置
3. 页面解析规则：列表选择器、正文选择器、附件选择器、发布时间选择器
4. 抓取频率和超时配置
5. 站点级限流与重试配置

#### 4.2.2 采集编排层

职责：

1. 定时触发栏目抓取
2. 将发现、抓取、解析、下载、入库任务拆分为异步阶段
3. 记录任务状态、失败原因和重试次数

建议能力：

1. 定时调度器
2. 任务队列
3. 任务状态机
4. 失败重试与死信机制

首期可用 Spring Scheduler + 数据库任务表实现。
后续可升级到 RabbitMQ 或 Redis Stream。

#### 4.2.3 内容治理层

职责：

1. 正文抽取
2. 附件下载与解析
3. OCR 兜底
4. 内容去重
5. 版本识别
6. 元数据抽取
7. 自动分类和质量评分

建议输出：

1. 标准化正文文本
2. 结构化元数据
3. 去重结果
4. 版本关系
5. 入库建议

#### 4.2.4 审核发布层

职责：

1. 人工审核待入库内容
2. 修正标题、分类、标签、地区、有效期等字段
3. 决定是否发布到主知识库
4. 决定进入哪个知识库目录或主题空间

建议支持：

1. 待审核池
2. 批量通过/驳回
3. 相似内容提示
4. 来源回溯
5. 自动审核规则开关

#### 4.2.5 知识入库层

职责：

1. 生成或更新知识文档记录
2. 存储原始文件和清洗结果
3. 调用现有切片与向量化链路
4. 维护知识文档与采集来源的映射关系

这里应复用当前已有能力：

1. DocumentLoaderService
2. TextSplitterService
3. KnowledgeService
4. MultiVectorStoreService

#### 4.2.6 检索服务层

职责：

1. 为政策查询提供高质量检索
2. 为政策匹配提供结构化过滤与召回
3. 控制不同场景使用不同知识范围

建议支持的过滤维度：

1. 地区
2. 政策类型
3. 适用品类
4. 发布时间
5. 有效期
6. 来源站点
7. 内容级别

## 5. 核心数据流

### 5.1 采集到入库主流程

1. 调度器按栏目配置创建 crawl_job
2. 栏目采集任务抓取列表页，发现文章链接和附件链接
3. 为每个候选内容创建 crawl_job_item
4. 内容抓取任务下载 HTML 或附件，并保存原始文件到 MinIO
5. 解析任务抽取正文、发布时间、来源、附件信息等元数据
6. 去重与版本任务判断是否为新内容、重复内容或旧文更新
7. 分类评分任务给出政策类型、地区、可信度和入库建议
8. 内容进入待审核池或自动审核通过
9. 发布任务生成 knowledge_document 并触发切片与 embedding
10. 检索服务对外提供给政策查询和政策匹配使用

### 5.2 主站使用流程

#### 政策查询

1. 用户提问
2. 系统按 query 类型确定检索范围
3. 从主知识库和必要的补充知识库中检索
4. 按地区、时效、类型过滤结果
5. 返回答案并带来源说明

#### 政策匹配

1. 用户提交个人或设备条件
2. 匹配引擎把条件映射为地区、品类、预算、身份类型等结构化约束
3. 先按结构化字段过滤候选政策
4. 再用向量召回和 rerank 精排
5. 输出适配政策与匹配理由

因此，政策匹配比政策查询更依赖结构化元数据质量，进一步说明必须建设治理层，而不是只做爬取。

## 7. 去重、版本与发布策略

### 7.1 去重策略

建议分三层：

1. URL 去重：相同 source_url 默认视为同一来源内容。
2. 内容指纹去重：对标准化文本生成哈希，避免换链接重复入库。
3. 标题加发布日期近似去重：防止列表页和正文页、附件页之间产生重复。

### 7.2 版本策略

当满足以下情况时，视为新版本而不是新文档：

1. 来源 URL 相同但正文哈希变化
2. 标题高度相似且发布时间一致，但正文有显著差异
3. 同一通知后续出现“修订版”“更新版”“补充通知”

策略建议：

1. 原始版本保留
2. 主知识库默认只暴露当前有效版本
3. 匹配场景默认使用最新有效版本

### 7.3 发布策略

建议分三级：

1. 自动丢弃：活动宣传、无正文、低质量页面
2. 待审核：边界不清、规则不确定、相似度冲突内容
3. 自动发布：高置信度政策通知、实施细则、附件正文完整内容

## 8. 对现有项目的落地建议

### 8.1 可直接复用的现有能力

当前仓库中以下能力可直接复用：

1. KnowledgeService：知识文档上传、状态管理、重入库
2. DocumentLoaderService：文档解析、PDF OCR 兜底
3. TextSplitterService：切片
4. MultiVectorStoreService：多 embedding 模型向量存储
5. AdminKnowledgeController：管理端知识库接口
6. scripts/scrape_shandong.py：现有站点抓取经验与规则基础

### 8.2 建议新增的后端模块

建议新增包结构：

1. crawler：站点适配、栏目抓取、正文提取
2. ingestion：采集任务编排、任务状态管理
3. curation：去重、分类、评分、审核建议
4. source：数据源配置和任务实体、仓储、服务

### 8.3 建议新增的管理端页面能力

管理员控制台建议在知识库模块旁增加：

1. 数据源管理
2. 栏目管理
3. 采集任务中心
4. 待审核内容池
5. 采集日志与失败重试

## 9. 分阶段实施建议

### 阶段 1：最小闭环

目标：先让山东省商务厅等少量来源实现自动采集并进入审核池。

范围建议：

1. source_site 和 source_section 配置表
2. Spring Scheduler 定时抓取
3. 抓取结果写入 source_content
4. 人工审核后调用现有 KnowledgeService 入库
5. 管理端增加数据源和待审核页面

这一阶段在表现形式上接近方案一，但实现结构应为方案二的前置骨架。

### 阶段 2：治理增强

目标：让系统具备可持续扩展能力。

范围建议：

1. 附件下载与解析异步化
2. 去重和版本识别
3. 质量评分与自动分类
4. 任务失败重试和监控指标
5. 自动发布规则

### 阶段 3：服务主站业务

目标：为政策查询和政策匹配提供更高质量支持。

范围建议：

1. 检索前结构化过滤
2. 匹配场景特定知识视图
3. 来源解释与时效提示
4. 按地区和品类的知识域划分
5. 政策变更提醒与订阅能力

## 10. 初步风险与注意事项

### 10.1 技术风险

1. 不同政府网站结构差异大，通用规则抽象要适度，避免一开始过度设计。
2. 附件型政策较多，OCR 和文档解析失败率要重点监控。
3. 如果没有版本控制，后续知识库会很快积累大量重复内容。

### 10.2 产品风险

1. 如果把活动新闻与政策正文混入同一个主知识库，会直接影响政策查询和匹配质量。
2. 如果没有人工审核兜底，初期自动分类容易引入脏数据。
3. 如果没有来源解释，用户对答案可信度会存疑。

### 10.3 运维风险

1. 多站点采集必须限流，避免对公开网站造成过高压力。
2. 采集失败、解析失败、入库失败必须分阶段可观测，否则很难运维。
3. 原始数据应保留到对象存储，避免规则调整后无法重处理。

---

## 11. 当前建议结论

针对“主站政策查询和政策匹配都将依赖知识库，且知识来源会持续扩展到多个网站”的前提，建议如下：

1. 选择方案二作为长期架构方向。
2. 首期按方案一的交付形态快速落地，但底层数据模型和任务流转必须按方案二预留扩展点。
3. 不要把自动采集直接接入向量库，而应先进入内容治理和审核层。
4. 把知识库从“上传文档功能”升级为“多源政策内容供给平台”。

这是更适合当前业务发展阶段的路线。
