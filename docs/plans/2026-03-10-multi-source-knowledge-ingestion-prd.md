## 1. 项目概述

### 1.1 背景

当前系统知识库主要依赖管理员手动上传文档，存在以下问题：

1. 知识来源有限，覆盖不完整，无法稳定支撑主站的政策查询与政策匹配。
2. 数据更新依赖人工收集，时效性弱。
3. 现有知识库更偏“文档存储”，尚未形成“多源内容供给平台”。
4. 后续政策来源不会局限于单一网站，而会扩展到省级厅局网站、地市政府网站、专题栏目、公告附件等多个公开来源。

因此需要建设一个多源政策知识库采集与服务化系统，实现对公开网站政策内容的采集、治理、审核、发布和入库，作为主站政策查询与政策匹配的核心底座。

首期不新增独立知识库产品线，不重做现有知识库管理模式，而是在现有“管理员-控制台-知识库”页面内增加一种新的知识来源获取方式：网站链接采集。

### 1.2 产品目标

本项目目标如下：

1. 在现有知识库能力基础上，支持新增“网站链接采集”这一知识来源方式。
2. 支持文章正文、PDF、DOC、DOCX 等附件型政策内容自动采集与解析。
3. 支持对采集结果进行去重、分类、结构化抽取、版本识别与质量评分。
4. 提供管理员预览、审核、确认入库能力，确保主知识库内容质量。
5. 让政策查询与政策匹配都能使用统一且可解释的知识底座。
6. 为后续扩展更多站点、更多规则和更高自动化能力预留架构空间。

### 1.3 方案结论

本项目采用以下总体策略：

1. 长期架构选择“事件驱动的内容采集 ETL 平台”。
2. 首期产品交付采用“单网站链接采集 + 人工审核入库”的最小闭环。
3. 首期重点是复用现有知识库页面、现有知识文档存储能力和现有知识库接口。
4. 所有首期能力在数据模型和流程设计上必须兼容后续事件驱动扩展。

### 1.4 首期实施边界

首期仅以以下网站栏目作为示例和落地对象：

1. http://commerce.shandong.gov.cn/col/col352659/index.html

首期目标不是一次性完成通用多站点平台，而是先在该网站上打通知识库扩充闭环：

1. 管理员在知识库页面录入网站链接。
2. 分析网页页面结构，按照不同的分类，分别爬取各自分类下的页面及其相关政策内容。
3. 系统将抓取结果作为待入库文档展示在现有知识库页面中。
4. 管理员通过预览判断内容是否可用，并决定是否正式入库。

## 3. 业务范围

### 3.1 范围内

1. 在现有知识库页面中新增网站链接导入能力
2. 网站正文与附件采集
3. 采集内容标准化治理
4. 待入库文档预览与确认入库流程
5. 与现有知识库管理能力集成
6. 为政策查询与政策匹配提供结构化知识供给

### 3.2 暂不纳入

1. 非公开网页或需要登录的网站采集
2. 视频、音频类复杂媒体内容识别
3. 首期通用化多站点规则编辑后台
4. 用户级个性化订阅推送
5. 完整工作流引擎替换

## 4. 核心业务场景

### 4.1 场景一：管理员通过网站链接发起知识扩充

管理员在“管理员-控制台-知识库”页面点击新增来源，选择“网站链接导入”，输入指定栏目链接后发起采集任务。

### 4.2 场景二：附件型政策正文自动入库前治理

系统抓取到 PDF 或 DOC 附件后，自动下载并解析正文。若文本提取失败，则尝试 OCR。完成后进入去重、分类和元数据抽取流程。

### 4.3 场景三：管理员预览后确认入库

管理员在知识库页面查看待入库文档，预览正文或附件内容，修正必要字段后，确认入库。系统随后生成知识文档记录并触发切片和向量化。

### 4.4 场景四：政策查询使用多源知识

用户在主站发起政策咨询时，系统从主知识库中检索高质量政策内容，并基于来源、时间、地区等信息生成可信回答。

### 4.5 场景五：政策匹配依赖结构化知识

用户输入地区、品类、设备或身份条件后，系统基于结构化元数据先做过滤，再做向量召回和排序，输出更准确的匹配结果。

## 5. 产品设计原则


重要！！！：确保爬虫脚本获得的信息的质量足够高才可以入库，确保爬取的都是政策文件的核心完整内容，必要时增加数据清洗/正则表达式等方法来提高爬虫脚本的质量。


1. 自动化优先，但保留人工审核兜底。
2. 先治理后入库，避免把脏数据直接送入向量库。
3. 原始内容必须保留，支持后续回放和重处理。
4. 所有关键内容必须可追溯来源。
5. 首期必须复用现有知识库页面、列表、预览、状态和上传后处理能力。
6. 检索和匹配共享统一知识底座，但允许按场景过滤不同内容范围。

## 8. 核心流程设计

### 8.1 网站链接导入流程

1. 管理员进入知识库页面。
2. 点击新增来源。
3. 选择网站链接导入。
4. 输入网站链接并选择目标文件夹。
5. 提交后系统创建采集任务并发送开始通知。

### 8.2 采集发布流程

1. 系统抓取目标网页内容。
2. 发现文章与附件链接。
3. 下载原始内容并归档。
4. 抽取正文、元数据和附件信息。
5. 执行去重和质量评分。
6. 内容进入待入库列表。
7. 管理员预览并确认是否入库。
8. 系统复用现有知识库入库链路完成切片与向量化。
9. 主站业务消费最新知识内容。

### 8.3 异常处理流程

1. 若页面抓取失败，则记录为采集失败并通知管理员。
2. 若附件下载失败，则标记该附件失败但不阻塞整条内容进入待入库区。
3. 若正文抽取失败，则保留原始内容并标记失败。
4. 若入库失败，则保留待入库状态并支持人工重试。

## 10. 接口需求

### 10.1 首期接口设计原则

1. 优先复用现有 /api/admin/knowledge 下的知识库接口。
2. 网站导入产生的候选文档在展示、预览、状态流转、最终入库上应尽量复用现有知识库接口语义。
3. 首期仅为“提交网站链接采集”和“待入库确认”补充最小必要接口。

### 10.2 复用的现有接口

1. GET /api/admin/knowledge/folders
2. POST /api/admin/knowledge/folders
3. GET /api/admin/knowledge/documents
4. GET /api/admin/knowledge/documents/{id}
5. GET /api/admin/knowledge/documents/{id}/preview
6. GET /api/admin/knowledge/documents/{id}/chunks
7. POST /api/admin/knowledge/documents/{id}/reingest
8. GET /api/admin/knowledge/config
9. GET /api/admin/knowledge/embedding-models

### 10.3 首期新增最小接口

1. POST /api/admin/knowledge/url-imports
2. GET /api/admin/knowledge/url-imports
3. GET /api/admin/knowledge/url-imports/{id}
4. POST /api/admin/knowledge/url-imports/{id}/confirm
5. POST /api/admin/knowledge/url-imports/{id}/reject

说明：

1. 这些接口只负责“网站链接采集”和“待入库确认”流程。
2. 真正落库后仍进入现有 knowledge_documents 体系。

### 10.4 主站知识服务接口要求

现有知识服务无需在首期全面改造，但必须满足以下产品要求：

1. 政策查询可按来源、地区、类型、时效过滤。
2. 政策匹配可使用结构化字段进行候选筛选。
3. 返回结果可附带来源链接和发布时间。

## 11. 规则与策略要求

### 11.1 去重规则

1. 相同来源 URL 默认视为同一内容来源。
2. 相同正文哈希视为重复内容。
3. 相近标题加相同发布时间视为疑似重复，需人工确认。
4. 首期若检测到重复内容，不直接入库，而是进入待确认状态。

### 11.2 版本规则

1. 同一来源 URL 内容更新时建立新版本。
2. 标题相近且正文显著变化时可建立版本关系。
3. 主站默认使用当前有效版本。
4. 首期版本规则可以先弱化为“保留来源映射 + 避免重复入库”。

### 11.3 自动发布规则

首期默认不启用完全自动发布。

仅允许以下内容在后续阶段评估自动发布：

1. 标题明确为通知、实施细则、工作方案的正式政策内容
2. 正文完整且发布时间明确
3. 质量评分达到高等级
4. 不存在重复或版本冲突

## 12. 非功能性需求

### 12.1 安全性

| 需求 | 说明 |
|------|------|
| NFR-1 | 所有后台采集与审核接口必须要求管理员权限 |
| NFR-2 | 原始抓取内容仅后台可见，前台不可直接访问未审核数据 |
| NFR-3 | 采集来源必须为公开可访问网站，不支持绕过鉴权抓取 |

### 12.2 性能与稳定性

| 需求 | 说明 |
|------|------|
| NFR-4 | 单次网站链接提交后应快速返回受理结果，不阻塞管理员页面操作 |
| NFR-5 | 采集失败必须可重试，且不影响现有知识库功能 |
| NFR-6 | 入库失败时必须保留中间状态，不可丢失数据 |

### 12.3 可观测性

| 需求 | 说明 |
|------|------|
| NFR-7 | 必须记录任务状态、失败原因和处理时间 |
| NFR-8 | 必须可以查看每条采集内容的来源、处理链路和审核结果 |
| NFR-9 | 首期至少要能在知识库页面中看到网站导入任务的最新处理状态 |

### 12.4 可扩展性

| 需求 | 说明 |
|------|------|
| NFR-10 | 新增网站来源方式不应破坏现有手动上传流程 |
| NFR-11 | 后续可从单链接采集平滑扩展到多站点调度采集 |
| NFR-12 | 数据模型需支持多站点、多版本、多附件 |

## 13. 验收标准

### 13.1 首期验收范围

首期以 http://commerce.shandong.gov.cn/col/col352659/index.html 为示例实施对象，完成以下闭环：

1. 管理员可在现有知识库页面新增网站链接来源。
2. 系统可抓取该网站页面内容及相关附件。
3. 系统可将抓取结果作为待入库文档展示在知识库页面。
4. 管理员可预览待入库内容并决定是否入库。
5. 确认入库后内容进入现有知识库，并可被政策查询使用。
6. 整个流程不破坏现有手动上传、预览、删除、重入库等能力。

### 13.2 二期验收方向

1. 支持版本识别。
2. 支持更完善的质量评分。
3. 支持批量审核与批量发布。
4. 支持为政策匹配提供结构化过滤数据。
5. 支持任务重试、统计报表和健康监控。

## 14. 实施优先级建议

### P0

2. 版本识别
3. 质量评分
4. 批量处理
5. 更完善的任务状态统计

### P2

1. 自动发布规则
2. 多站点统一配置
3. 主站政策匹配专用检索视图
4. 政策变化提醒能力
