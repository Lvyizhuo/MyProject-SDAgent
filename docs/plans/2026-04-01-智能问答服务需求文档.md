# 智能问答服务需求文档

## 1. 文档信息

- 文档名称：智能问答服务需求文档
- 项目：山东省智能政策咨询助手（MyProject-SDAgent）
- 编写日期：2026-04-01
- 适用范围：后端问答主链路（`/api/chat`、`ChatService`、Advisor 链、工具调用、Redis 会话记忆）

## 2. 背景与目标

当前问答链路存在“第一阶段职责膨胀”和“工具执行前置”问题：

1. 第一阶段（快速通道）除了 FastPath 和 QA 语义缓存外，还执行了 RAG 预检索与高置信直答，导致链路变重。
2. 第一阶段末尾提前触发了 `webSearch`/`calculateSubsidy` 直连调用，和意图分类器 + Agent 的二阶段职责重叠。
3. 会话历史压缩策略目标是“滚动摘要 + 最近两条消息”，但默认值仍保留“最近四条”的后备配置。
4. FastPath 规则覆盖不足，未覆盖高频政策咨询问题（补贴比例、材料、流程、到账时间、地区补贴）。
5. 工具调用提示词与结构化会话事实仍有优化空间，影响工具调用精度与可解释性。

目标：将第一阶段收敛为“纯快速响应通道（不依赖 LLM）”，将工具调用决策统一交给第二阶段 Agent，并补全 FastPath 与工具调用提示词体系。

## 3. 当前实现分析（基于代码）

### 3.1 入口与消息落库

- `ChatController.chat()` 中已完成：
  - 生成/补齐 `conversationId`
  - 保存用户消息到会话存储
  - 调用 `ChatService.chat()`
  - 保存助手回复

### 3.2 第一阶段现状（ChatService）

在 `ChatService.chat()` 中，当前顺序为：

1. `sessionFactCacheService.mergeFacts()`：读取并合并结构化事实（Redis）。
2. `fastPathService.tryDirectAnswer()`：FastPath 快速规则（当前仅问候、能力说明两类）。
3. `questionSemanticCacheService.lookup()`：QA 语义缓存命中后返回。
4. `ragPrefetchService.prefetch()`：执行 RAG 预检索，并可能触发“高置信直答（仍进入 LLM）”。
5. 执行计划生成 + 意图分类后，存在后端直连 `webSearch` 与 `calculateSubsidy` 的预执行逻辑。

结论：第一阶段目前并非“纯快速通道”，仍包含 RAG 和工具预执行，偏离设计目标。

### 3.3 第二阶段现状（LLM + Advisor）

- 二阶段通过 `executeChatWithFallback()` + `dynamicChatClientFactory.create()` + `executePrompt()` 执行。
- Advisor 链已包含：Security、Memory、ReReading、QuestionAnswer、Logging。
- 第二次 Redis 查询由 `MessageChatMemoryAdvisor` 完成，主要用于会话历史注入。

结论：二阶段具备完整的 Agent/工具/检索能力，应承接所有需要 LLM 推理或工具决策的动作。

### 3.4 Redis 历史记忆现状

- `application.yml` 已配置 `app.advisor.memory.summary-keep-messages: 2`。
- 但 `RedisChatMemory` 与 `ChatClientConfig` 中默认值仍为 4（配置缺失时会回退到 4）。

结论：需统一默认行为，避免环境差异导致“最近消息窗口”不一致。

### 3.5 工具与提示词现状

- `SubsidyCalculatorTool` 的 `@Description` 为单段文本，参数约束与调用时机表达不够结构化。
- 默认系统提示词较简洁，未明确“结构化信息复用、品牌规范化、搜索关键词构造、补贴计算前置校验”等规则。
- `SessionFactCacheService.SessionFacts` 当前以设备型号/品类/地区/价格为主，缺少更细粒度商品结构字段。
- `ToolIntentClassifier` 已有基础过滤，但商品型号识别与搜索拦截策略仍可细化。

## 4. 需求范围

- 包含：问答主链路阶段划分、FastPath、RAG 预检索策略、工具调用时机、Redis 会话记忆窗口、工具提示词与意图分类增强。
- 不包含：前端 UI 改造、数据库 schema 强制迁移（若仅新增缓存字段可不落库）、第三方模型供应商切换。

## 5. 详细需求

### R1 第一阶段移除 RAG 预检索

#### 目标

第一阶段仅保留“无需 LLM 的快速响应能力”，不再执行 RAG 预检索与 RAG 高置信直答。

#### 功能要求

1. 移除 `ChatService.chat()` 中第一阶段 `ragPrefetchService.prefetch()` 相关调用。
2. 移除 `shouldReturnDirectRagAnswer()` 触发的“预检索后再走 LLM 直答”分支。
3. 第一阶段完成后，未命中直接进入第二阶段 Agent/LLM。

#### 验收标准

1. 第一阶段日志不再出现“RAG 预检索完成”与“RAG预检索高置信直答”。
2. 第一阶段平均耗时下降（基线对比至少下降 20%，以压测或采样日志为准）。
3. 同样输入下，第一阶段不会访问向量检索服务。

### R2 QA 语义缓存职责前移，第二阶段不重复查

#### 目标

QA 语义缓存只在第一阶段快速通道中查询，第二阶段仅使用会话历史注入上下文。

#### 功能要求

1. 明确 `QuestionSemanticCacheService.lookup()` 仅在第一阶段调用。
2. 第二阶段 Advisor/Prompt 组装不新增 QA 语义缓存读取逻辑。
3. 文档与注释中明确职责边界：
   - 第一阶段：FastPath + QA 语义缓存。
   - 第二阶段：会话历史 + RAG + 工具 + LLM。

#### 验收标准

1. 全链路中 QA 语义缓存查询次数每请求最多 1 次。
2. 二阶段执行日志不出现 QA 缓存命中/查询相关记录。

### R3 移除第一阶段末尾的工具预调用

#### 目标

工具调用交由第二阶段 Agent 根据意图分类结果自主决策，不在第一阶段末尾提前调用。

#### 功能要求

1. 移除 `executeDirectWebSearchIfNeeded()`、`executeForcedPriceLookupIfNeeded()`、`executeDirectSubsidyIfNeeded()` 在第一阶段与二阶段交界处的预执行入口。
2. 保留工具意图分类器前置校验能力，但仅用于生成执行建议，不直接执行工具。
3. `buildExecutionPrompt()` 中工具输入信息来源改为：
   - 用户输入
   - 会话事实缓存
   - Agent 规划补问结果

#### 验收标准

1. 在进入 `executeChatWithFallback()` 前不发生外部工具调用。
2. 工具调用日志应仅出现在 Agent 执行阶段（LLM 工具回调阶段）。

### R4 Redis 会话历史窗口统一为“最近两条”

#### 目标

统一策略为“滚动摘要 + 最近两条消息”，并确保默认值与配置值一致。

#### 功能要求

1. 将 `RedisChatMemory.DEFAULT_SUMMARY_KEEP_MESSAGES` 调整为 2。
2.       summary-trigger-messages: 8  改为 6
3. 将 `ChatClientConfig` 中 `@Value("${app.advisor.memory.summary-keep-messages:4}")` 默认值调整为 2。
4. 保持 `application.yml` 的 `summary-keep-messages: 2`。

#### 验收标准

1. 在未显式配置该参数的环境中，仍默认保留最近 2 条消息。
2. 触发摘要后 Redis 中明细消息条数为 2（不含摘要）。

### R5 FastPath 规则增强

#### 目标

将高频、确定性问题在第一阶段直接回答，减少不必要的 LLM 调用。

#### 功能要求

1. 保留现有规则：问候、能力询问。
2. 新增规则：
   - 补贴比例问题
   - 申请材料问题
   - 申请流程问题
   - 到账时效问题
   - 基于会话事实（城市）的“本地是否有补贴”问题
3. 规则匹配函数需保持可维护性：独立私有方法 + `normalize()` 统一预处理。
4. 涉及地区规则时优先使用结构化事实（如 `city` 字段），无事实时不误判。

#### 验收标准

1. 新增规则单元测试覆盖每一类问题（命中与不命中样例）。
2. 命中 FastPath 时不触发 LLM、RAG、工具调用。
3. 命中回答内容符合政策助手语气，包含可执行下一步引导。

### R6 工具调用提示词与结构化信息增强

#### 目标

提升工具调用准确率、参数完整率与可解释性，减少模糊查询和无效调用。

#### 功能要求

1. **工具描述增强（SubsidyCalculatorTool）**
   - 使用结构化 `@Description`（用途、必填参数、补贴规则、调用时机、使用约束）。
   - 明确“参数缺失必须追问，禁止猜测补全”。

2. **系统提示词增强（默认提示词与管理员配置模板）**
   - 增加工具调用优化规则：上下文复用、商品标准化提取、品牌规范化、搜索关键词构造、补贴计算前置校验。
   - 明确“模糊价格/模糊类型不得直接调用补贴计算”。

3. **会话事实实体增强（SessionFact）**
   - 在不破坏兼容的前提下扩展字段：品牌、系列、型号、规格、类型、年份、政策类型、补贴比例等。
   - 增加商品型号正则提取与品牌标准化方法。

4. **意图分类器增强（ToolIntentClassifier）**
   - 增加“明确型号识别”能力（型号、规格、年份款）。
   - `webSearch` 分类逻辑改为：
     - 有明确型号：允许。
     - 无明确型号：拦截并追问“品牌+型号”。

#### 验收标准

1. 工具调用参数缺失率下降（以日志统计，目标下降 30% 以上）。
2. `webSearch` 宽泛问题被正确拦截并补问，明确型号问题可放行。
3. 补贴计算前必须具备“明确商品类型 + 明确数字价格”。

## 6. 非功能需求

1. 可观测性：保留关键阶段日志，新增阶段标签（`fastPath`、`qaCache`、`agentPhase`）。
2. 性能：第一阶段不依赖外部检索和工具调用，P95 响应时延应明显下降。
3. 兼容性：不改变外部 API 协议（请求/响应 JSON 结构保持不变）。
4. 可回滚性：通过配置开关支持分阶段灰度（如 FastPath 扩展开关、工具提示词版本开关）。

## 7. 测试要求

### 7.1 单元测试

1. `FastPathService`：新增规则覆盖（命中/不命中）。
2. `ToolIntentClassifier`：型号识别、宽泛查询拦截、补问文案。
3. `SessionFactCacheService`：新增结构化字段提取、品牌标准化。

### 7.2 集成测试

1. 第一阶段命中 FastPath：验证不触发 LLM/RAG/工具。
2. 第一阶段命中 QA 语义缓存：验证直接返回。
3. 第一阶段未命中：验证直接进入第二阶段 Agent。
4. 第二阶段工具调用：验证由 Agent 决策触发。

### 7.3 回归测试

1. 现有政策问答准确率不下降。
2. 会话历史上下文连续性不下降。
3. 语义缓存命中率与误命中率可控。

## 8. 发布与风险控制

### 8.1 发布步骤建议

1. 先发布第一阶段收敛（R1/R3）。
2. 再发布 FastPath 扩展（R5）。
3. 最后发布工具提示词与结构化增强（R6）。

### 8.2 主要风险

1. 移除预检索后，部分“短政策问答”可能更依赖二阶段 RAG，需关注时延。
2. 工具预调用移除后，极少数场景首次回答可能从“直接结果”变为“先澄清再调用”。
3. SessionFact 字段扩展需注意旧缓存反序列化兼容。

### 8.3 回滚策略

1. 保留配置开关以便快速回退到旧链路。
2. 保留原有提示词模板版本，支持按配置切换。
