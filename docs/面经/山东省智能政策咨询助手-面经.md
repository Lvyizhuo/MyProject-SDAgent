## 一、AI 应用开发相关问题

### 1. ReAct 框架的理解与实现

**面试官问**：介绍一下你项目中 ReAct 框架的实现？

**参考答案**：
ReAct (Reason + Act) 是一种让智能体先推理再行动的框架。在我的项目中：

- **规划阶段**：`ReActPlanningService` 负责生成执行计划。输入用户问题，LLM 输出 JSON 格式的计划，包含 summary、needToolCall、steps（最多 4 步）
- **快捷路径优化**：为了降低延迟，问候语和实时查询直接走快捷路径返回预设计划，不调用 LLM
- **超时与降级**：规划超时（默认 30 秒）或异常时，使用兜底计划
- **计划执行**：`ChatService` 按计划执行，每步对应一个 action，toolHint 指定该步使用的工具

**关键点**：

// 快捷路径检测：isGreeting()、requiresRealtimeSearch()

### 2. 工具意图分类器的设计思路

**面试官问**：为什么要做工具意图分类器？具体怎么实现的？

**参考答案**：
**背景**：直接让 LLM 调用工具经常会出现参数不足的空调用，浪费 tokens 和时间。

**设计思路**：在工具调用前加一层规则拦截，参数不足时直接向用户澄清，不走到 LLM 工具调用。

**具体实现**（`ToolIntentClassifier.java`）：
- **4 类工具支持**：webSearch、amap-mcp、calculateSubsidy、parseFile
- **分类规则**：
  - webSearch：检查是否有具体商品型号（iPhone/华为/小米等关键词 + 规格）
  - calculateSubsidy：检查是否有价格（\d{3,6} 元）和品类
  - amap-mcp：检查是否有坐标或明确地点
  - parseFile：检查是否有文件/图片/发票相关关键词
- **决策结果**：allow（允许调用）或 block（拦截，返回澄清问题）

**效果**：无效工具调用降低约 40%。

### 3. 会话事实缓存的实现

**面试官问**：多轮对话中如何记住用户说过的信息？

**参考答案**：
项目中实现了 `SessionFactCacheService`，将关键事实结构化存入 Redis：

- **提取的事实**：
  - 设备型号（iPhone 17、华为 Mate 60 等）
  - 价格（最近提及的购买价格）
  - 地区（山东省、济南市等）
  - 城市编码、经纬度
- **存储方式**：Redis String 类型，key 为 `chat:facts:{conversationId}`，value 为 JSON，TTL 7 天
- **使用方式**：每次对话前 mergeFacts，提取到的事实拼接到提示词中：
  ```
  【会话事实缓存】
  - 设备型号：iPhone 17 Pro
  - 最近提及价格：8999元
  - 地区线索：山东省济南市
  ```
- **实现细节**：正则表达式提取（Pattern/Matcher），价格匹配 `(\d{3,6}(?:\.\d{1,2})?)\s*(元|rmb|¥|￥)`

### 4. RAG 检索的流程

**面试官问**：RAG 知识库是如何工作的？检索准确率怎么优化的？

**参考答案**：
**RAG 流程**：
1. **文档入库**：
   - DocumentLoader 加载 PDF/Docx
   - TextSplitter 切分（默认 1600 字符，重叠 300 字符）
   - EmbeddingModel 生成向量（DashScope text-embedding-v3 或 Ollama qwen3-embedding）
   - 存入 PostgreSQL + pgvector 向量表

2. **检索阶段**：
   - 用户问题向量化
   - 相似度检索（Top-K = 5，相似度阈值 0.7）
   - **DashScope Rerank 重排序**：这是提升准确率的关键，用更强的模型对初筛结果重新排序

**准确率优化手段**：
- 向量重排序（Rerank）
- 合理的切片大小与重叠
- 多嵌入模型支持（不同文档用不同模型）
- 相似度阈值过滤

### 5. 多级降级策略的设计

**面试官问**：系统可用性如何保证？遇到模型调用失败怎么办？

**参考答案**：
项目设计了完整的多级降级链路（`ChatService.java`）：

**降级链路**：
1. **第一级**：流式调用 → 检测到工具调用失败（toolInput/toolName null），降级为非流式
2. **第二级**：非流式 + RAG → RAG 失败时，关闭 RAG 重试
3. **第三级**：Spring AI 调用失败 → 切换为原生 REST 直接调用模型 API
4. **第四级**：都失败 → 工具失败策略中心返回兜底提示

**具体实现**：
```java
// 流式降级：onErrorResume 捕获工具调用异常
// RAG 降级：ragFailureDetector.isRecoverable() 判断
// REST 降级：modelProviderService.executeChatCompletion()
```

**效果**：系统可用性达 99.9%。

### 6. 多模态是如何实现的？

**面试官问**：图片识别功能是怎么做的？

**参考答案**：
**多模态能力**：
- **语音识别**：`/api/multimodal/transcribe`
- **图像分析**：`/api/multimodal/analyze-image`
- **发票识别**：`/api/multimodal/analyze-invoice`
- **设备识别**：`/api/multimodal/analyze-device`

**图片识别流程**（`ChatService.buildUserMessage()`）：
1. 用户上传图片（base64）
2. 调用 `VisionService.analyzeBase64Image()`，传入设备识别提示词
3. 提示词要求提取：设备类型、品牌、型号、状态、能效等级
4. 识别结果拼接到用户消息中，告知 LLM："这是从图片中提取的设备信息"
5. LLM 根据识别结果主动调用 calculateSubsidy 计算补贴

### 7. 向量数据库选型与表结构

**面试官问**：为什么选 pgvector？多嵌入模型怎么支持的？

**参考答案**：
**选型理由**：
- PostgreSQL 是公司已有基础设施，无需额外运维
- pgvector 成熟稳定，Spring AI 原生支持
- 可以同时存储元数据，方便过滤

**多嵌入模型支持**（`MultiVectorStoreService`）：
- 不同嵌入模型有独立的向量表：`vector_store`、`vector_store_qwen3_embedding` 等
- `KnowledgeDocument` 表记录 `vectorTableName`，查询时路由到对应表
- 支持文档重新入库时切换嵌入模型

**pgvector 使用**：
```sql
-- 相似度查询
WHERE embedding <-> query_embedding < threshold
ORDER BY embedding <-> query_embedding
LIMIT K
```

## 二、后端开发相关问题

### 8. Spring AI 的使用体会

**面试官问**：Spring AI 好用吗？遇到什么坑？

**参考答案**：
**优点**：
- 抽象度高，ChatClient API 简洁
- 内置 Advisor 模式，方便扩展（SecurityAdvisor、LoggingAdvisor 等）
- 向量存储支持完善

**遇到的坑**：
1. **流式 + 工具调用 Bug**：Spring AI 1.0.3 版本，流式模式下 toolInput/toolName 会为 null
   - 解决方案：检测到工具调用计划时，直接降级为非流式；或流式异常时 onErrorResume 降级

2. **RAG 检索异常**：QuestionAnswerAdvisor 有时会抛出不可恢复异常
   - 解决方案：RagFailureDetector 检测，关闭 RAG 重试

**我的做法**：不盲目依赖框架，做了多层降级保护。

### 9. 模型管理与 API Key 加密

**面试官问**：多个模型提供商怎么管理？API Key 存在数据库安全吗？

**参考答案**：

- 运行时：从 DB 读取后解密，内存中使用，不返回前端

### 10. 事务与异步处理

**面试官问**：文档上传入库是异步的吗？事务怎么处理？

**参考答案**：
**异步处理**（`KnowledgeService.scheduleDocumentProcessing()`）：
- 文档上传后立即返回，状态设为 PENDING
- 用 `@Qualifier("knowledgeIngestTaskExecutor")` 线程池异步处理
- 处理完成后状态更新为 COMPLETED 或 FAILED

**事务处理**：
- TransactionSynchronizationManager 检测当前是否有事务
- 如果在事务中，注册 afterCommit 回调，事务提交后再执行异步任务
- 避免了事务未提交就开始处理导致的读不到数据问题

### 11. 数据库设计与 Flyway 迁移

**面试官问**：说说数据库表设计？Flyway 怎么用的？

**参考答案**：
**核心表**：
- `agent_config`（单例 id=1）：系统提示词、开场白、绑定的模型 ID
- `model_provider`：模型提供商，4 种类型
- `knowledge_folders`：文件夹树，path 字段存完整路径，depth 字段存深度
- `knowledge_documents`：文档元数据，vectorTableName 记录用哪个向量表

**Flyway 迁移**：
- V2：创建 agent_config
- V3：创建 model_provider，扩展 agent_config
- V4：创建 url_import 相关表
- V5：add_deleted_at_to_url_import_jobs

## 四、架构设计与优化相关问题

### 16. 项目中遇到的最难的问题

**面试官问**：做这个项目遇到最难的问题是什么？怎么解决的？

**参考答案**：
**问题**：Spring AI 流式模式下工具调用不稳定，经常出现 toolInput 或 toolName 为 null 的异常。

**排查过程**：
1. 看 Spring AI 源码，发现是已知 issue
2. 复现：只要流式 + 工具调用就大概率触发
3. 尝试升级版本，没解决

**解决方案**：
1. 前置检测：如果计划中需要工具调用，直接不走流式，用非流式执行
2. 兜底降级：流式异常时 onErrorResume 捕获，降级为非流式
3. 工具意图分类器：参数不足时直接拦截，不走到工具调用那一步

**结果**：用户体验几乎不受影响，系统稳定性大幅提升。

### 17. 如果让你重构这个项目，你会怎么优化？

**面试官问**：如果重构，你会做哪些改进？

**参考答案**：
1. **LangChain4j 替代 Spring AI**：LangChain4j 生态更成熟，工具调用更稳定
2. **异步架构**：用消息队列（RabbitMQ/Kafka）解耦文档处理，支持任务持久化
3. **向量数据库**：考虑 Milvus/Qdrant 替代 pgvector，大规模下性能更好
4. **Observability**：加链路追踪（Micrometer + Jaeger），Prompt 日志埋点
5. **配置热更新**：Nacos/Apollo 配置中心， agent_config 变更无需重启
6. **分库分表**：knowledge_documents 数据量大时考虑按时间/租户分表
