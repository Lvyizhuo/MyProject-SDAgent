### 1. 介绍一下你的 RAG 项目

**面试官问**：介绍一下你做的 RAG 政策咨询助手？

**参考答案**（结合岗位往消费场景靠）：

**核心架构**：

- **多级降级**：流式 → 非流式 → REST 兜底，保证高可用

**数据指标**：
- 检索准确率提升 25%（Rerank）
- 系统可用性 99.9%
- 知识库文件 1000+ 份

### 2. RAG 有哪些常见的优化方法？

**参考答案**：
**检索侧优化**：
1. **Embedding 模型**：选任务适配的 Embedding（如 M3E、bge、DashScope text-embedding-v3）
2. **切片策略**：
   - 固定大小切片（1600 字符 + 300 重叠）
   - 语义切片（按段落/标题切分）
   - 父子索引（小切片检索，大切片给 LLM）
3. **向量重排序（Rerank）**：初筛 Top-K（如 20）→ Rerank 模型（如 bge-reranker、DashScope Rerank）→ 重排后 Top-N（如 5）
4. **混合检索**：BM25 关键词检索 + 向量检索，做 Reciprocal Rank Fusion（RRF）
5. **元数据过滤**：先按时间/地区/分类过滤，再做向量检索
6. **查询改写**：LLM 把用户问题改写为多个检索 query

**生成侧优化**：
1. **Prompt 工程**：RAG 结果如何组织、引用格式要求
2. **拒识判断**：检索结果不相关时，明确告知用户
3. **引用溯源**：在回答中标注来源，提升可信度

**我项目中的实践**：用了 Rerank + 合理切片 + 相似度阈值过滤，准确率提升 25%。

### 3. Agent 的理解与实践

**面试官问**：你项目中 ReAct 框架怎么实现的？

**参考答案**：
**ReAct = Reason + Act**，先推理再行动。

**我的实现**：
1. **规划阶段**（ReActPlanningService）：
   - 输入：用户问题
   - 输出：JSON 格式计划 {summary, needToolCall, steps}

   - 超时降级：30 秒超时，返回兜底计划

**想做但还没做的**（体现思考）：
- 更复杂的多步计划，执行中动态调整

- 反思（Reflection）环节，执行完总结

### 4. 你提到了工具意图分类器，能详细说说吗？

**面试官问**：工具意图分类器是规则还是模型做的？为什么不直接让 LLM 决定？

**参考答案**：
**为什么做**：
- LLM 直接调用工具经常参数不足，浪费 tokens 和时间
- 比如用户问"查一下价格"，LLM 可能就调用 webSearch，但没说查什么的价格
- 空调用，空调用，空调用，重要事情说三遍：这是**落地中非常实际的问题**

**我的实现**：规则为主，轻量高效。
- 4 类工具：webSearch、calculateSubsidy、amap-mcp、parseFile
- 分类逻辑：
  - **webSearch**：检查有没有具体商品型号（iPhone/华为/小米 + 规格关键词）
  - **calculateSubsidy**：检查有没有价格（\d{3,6} 元）和品类
  - **amap-mcp**：检查有没有坐标或明确地点
  - **parseFile**：检查有没有文件/图片/发票关键词
- 结果：allow（允许）或 block（拦截，返回澄清问题）

**为什么不用模型**：
- 规则已经能 cover 80% 场景，速度快，不花钱
- 模型当然可以做，但对这个场景有点 overkill
- 后续如果规则不够，可以升级为小模型分类器

**效果**：无效工具调用降 40%。

### 5. 关键信息抽取（KIE）你是怎么做的？

**面试官问**：会话事实缓存中的价格/地区/设备型号怎么提取的？

**参考答案**：
**背景**：多轮对话中，用户第一次说"iPhone 17 8999 元"，第二次问"补贴多少"，这时候需要记住之前的信息。

**我的简单实现**：正则表达式提取（SessionFactCacheService）
- **价格**：`(?<!\d)(\d{3,6}(?:\.\d{1,2})?)\s*(元|rmb|¥|￥)`
- **地区**：`([\p{IsHan}]{2,}(?:省|市|区|县))`
- **设备型号**：iPhone/华为/小米等品牌 + 型号关键词

**存储**：Redis String，key = `chat:facts:{conversationId}`，value = JSON，TTL 7 天

**使用**：提取到的事实拼接到提示词中：
```
【会话事实缓存】
- 设备型号：iPhone 17 Pro
- 最近提及价格：8999元
- 地区线索：山东省济南市
```

**进阶方案**（体现思考，往岗位靠）：
- 现在的正则比较 hardcode，后续可以用 **NER（命名实体识别）** 模型
- 比如用 UniversalNER、或 Qwen 等开源大模型做 NER
- 还可以用 **LLM 做结构化抽取**：prompt 让 LLM 输出 JSON，包含 price、region、device 字段

### 7. Token 压缩有了解吗？

**面试官问**：对话历史太长怎么办？有哪些 Token 压缩方法？

**参考答案**：
**常见方法**：
1. **滑动窗口**：只保留最近 N 轮对话
2. **摘要压缩**：用 LLM 把历史对话摘要成一段
3. **重要性筛选**：只保留系统消息、工具调用结果等重要消息

- 会话事实缓存也是一种压缩：把关键信息结构化，不用每次都从长历史中提取

**进阶**：
- 可以用 **LLMLingua** 等专门的 Token 压缩模型
- 消费场景下，用户历史可能很长，压缩很有必要

### 8. PEFT 有了解吗？

**面试官问**：微调过模型吗？PEFT 技术知道哪些？

**参考答案**（如果没实际微调过，就说了解+实践准备）：
**PEFT = Parameter-Efficient Fine-Tuning（参数高效微调）**，只训少量参数，不训全量。

**常见方法**：
1. **LoRA（Low-Rank Adaptation）**：
   - 在 Transformer Attention 层旁加低秩矩阵
   - 秩 r 通常 8-64，参数量很少
   - 推理时可以合并权重，不增加延迟
2. **QLoRA**：LoRA + 4 位量化，显存占用更低
3. **Adapter**：在层间加小适配器模块
4. **Prefix Tuning**：只训前缀向量
5. **P-Tuning v2**：连续 Prompt 微调

**我了解的实践**：
- 用 **Axolotl** 或 **LLaMA-Factory** 框架微调
- 消费场景下，可以用 Qwen/DeepSeek 等开源模型做 LoRA 微调，适配业务
- 数据方面：业务对话数据 + CoT（思维链）构造

**想做的**：如果有机会，想在业务数据上做 QLoRA 微调，让模型更贴合消费场景。

### 9. 强化学习对齐（RLHF）了解吗？

**面试官问**：PPO/DPO/GRPO 有了解吗？

**参考答案**：
**RLHF = Reinforcement Learning from Human Feedback**，用人类反馈对齐模型。

**主流方法**：
1. **PPO（Proximal Policy Optimization）**：
   - 经典 RL 算法
   - 三步：有监督微调（SFT）→ 奖励模型（RM）→ PPO 训练
   - 缺点：训练不稳定，Reward Hacking
2. **DPO（Direct Preference Optimization）**：
   - 直接从偏好数据学习，不需要 RM 和 PPO
   - 更简单，更稳定
   - 现在很流行
3. **IPO/GRPO**：DPO 的改进版
   - GRPO（Group Relative Preference Optimization）：分组相对偏好优化

**我的理解**：
- 消费场景下，需要模型回答友好、符合业务规范，RLHF/DPO 很有用
- 可以用 **trlX**、**TRL** 库实现
- 数据方面：构造偏好对（chosen vs rejected）

### 10. CoT（思维链）了解吗？

**面试官问**：思维链是什么？你项目中有用到吗？

**参考答案**：
**CoT = Chain-of-Thought**，让模型"先思考，再回答"，把推理过程写出来。

**常见方法**：
1. **Few-shot CoT**：给几个例子
   ```
   问：12+34=？
   答：10+30=40，2+4=6，40+6=46，所以答案是46
   ```
2. **Zero-shot CoT**：加一句"让我们一步步思考"
3. **Self-Consistency**：多条思维链，投票选答案
4. **Tree-of-Thought**：树状搜索

**我项目中的实践**：
- 提示词中要求模型先思考再行动
- ReAct 框架其实也是一种 CoT：Thought → Action → Observation
- 补贴计算时，会要求模型说明计算过程

**消费场景应用**：
- 商品推荐：让模型说明推荐理由
- 客服问答：让模型先分析问题，再给出解决方案
- 增加可解释性

## 二、机器学习/深度学习基础

### 11. Transformer 架构讲一下

**面试官问**：Transformer 的结构？Attention 机制？

**参考答案**：
**Transformer = Encoder + Decoder**（Encoder-Decoder 架构）

**Encoder**：
- 多层 Encoder Layer，每层 = Multi-Head Attention + Feed-Forward + LayerNorm + Residual
- 输入：Token Embedding + Position Embedding
- 输出：上下文表示

**Decoder**：
- 多层 Decoder Layer，每层 = Masked Multi-Head Attention + Cross-Attention + Feed-Forward
- Masked Attention：看不到后面的 Token（自回归）
- Cross-Attention：Attend to Encoder 输出

**Attention 公式**：
```
Attention(Q, K, V) = softmax(QK^T / sqrt(d_k)) V
```

**Multi-Head Attention**：
- 分成多个头，每个头学不同的注意力模式
- Concat 起来再线性变换

**Position Embedding**：
- 正弦余弦位置编码
- 或可学习位置编码

**常见变体**：
- **Decoder-only**：GPT 系列、LLaMA、Qwen
- **Encoder-only**：BERT
- **Encoder-Decoder**：T5、Whisper

### 13. 文本分类怎么做？

3. **模型选型**：中文用 Qwen、BERT-base-chinese、RoBERTa-wwm
4. **训练**：交叉熵损失、学习率调度、早停
5. **评估**：Accuracy、F1、Precision/Recall

**我项目中的相关实践**：
