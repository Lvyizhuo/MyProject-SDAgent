# 📑 产品需求规格说明书 (PRD) v1.1
## 1. 项目概述 (Project Overview)
本项目旨在开发一款基于 **Spring AI** 架构的**智能政策咨询助手**，初期覆盖**山东省及下辖地市**。系统利用大模型（LLM）结合 **RAG（检索增强生成）** 技术，通过 **MCP（Model Context Protocol）** 标准化集成外部工具，解决传统人工客服响应慢、政策理解门槛高、计算复杂的痛点。

项目核心目标是构建一个具备**安全监管（Advisors）**、**多模态交互**、**精准计算**能力的垂直领域智能体。
## 2. 用户角色与业务流程 (User Roles & Flow)
### 2.1 用户画像
- **C端用户（山东省居民）**：关注“能省多少钱”、“怎么操作”。需要极简交互。
- **管理员（运营人员）**：负责知识库 ETL（抽取转换加载）管理、Prompt 调优、以及 bad case 的日志审计。

### 2.2 核心业务流程（基于架构图优化）
1. **接入层**：用户通过微信小程序/H5 发起请求（文本/语音/图片）。
2. **前置拦截 (Advisors Pre-intercept)**：
    - **安全守卫 (Security)**：检测用户输入是否包含敏感词、攻击性指令。
    - **上下文增强**：自动携带历史会话记忆（ChatMemory）。
3. **核心处理 (Spring AI Service)**：
    - **意图识别**：调用 ChatModel（DeepSeek/通义）判断意图。
    - **RAG 检索**：若涉及政策，触发检索增强，从 VectorStore 召回切片。
    - **工具执行 (MCP/Tools)**：若涉及计算或实时信息，通过 MCP 协议调用本地工具或联网搜索。
4. **后置拦截 (Advisors Post-intercept)**：
    - **ReReading/纠错**：检查 AI 生成的答案是否准确引用了政策原文，防止幻觉。
    - **日志记录**：将交互数据写入审计日志。
5. **响应输出**：前端渲染流式文本、引用卡片或生成的 PDF 报告。

## 3. 功能需求详情 (Functional Requirements)
### 3.1 前端应用层 (Client & App Layer)

|**模块**|**功能点**|**基于架构图的技术支撑**|**优先级**|
|---|---|---|---|
|**对话交互**|引导式气泡|对应架构图“小程序”端，通过预设 Prompt 触发。|P0|
||多模态输入|支持图片上传，通过 ChatClient 调用多模态模型（如通义/Qwen-VL）。|P1|
||**政策报告生成**|**[新增]** 对话结束后，支持生成个性化政策解读 PDF 供下载。（对应架构图 Tools 中的 **PDF生成**）。|P2|
|**结果展示**|引用来源卡片|必须展示 RAG 检索到的元数据（文件名、页码）。|P0|

### 3.2 后端服务层 (Spring AI Core) [核心优化]
本模块严格对应架构图中的 **Spring AI 服务** 模块。
#### A. Advisors 智能拦截器系统
- **前置拦截 (Pre-Advisors)**：
    - **安全合规**：集成敏感词库，拦截涉及政治敏感或非业务相关的恶意提问。
    - **会话记忆 (ChatMemory)**：基于架构图的“内存存储/文件存储”，实现多轮对话上下文管理，确保用户说“那它呢？”时 AI 知道指代对象。
- **后置拦截 (Post-Advisors)**：
    - **ReReading 机制**：强制模型对生成的答案进行“再阅读”校验，确保金额、日期等关键信息与检索到的文档一致。
    - **结构化日志**：记录 Token 消耗、响应延迟、引用来源，用于后台分析。
#### B. 模型与编排 (ChatModel & ChatClient)
- **多模型路由**：
    - 支持动态切换模型。例如：简单闲聊使用 **DeepSeek-V3**（低成本），复杂逻辑推理使用 **通义千问 Max** 或 **DeepSeek-R1**。
    - 利用 Spring AI 的 `ChatClient` 统一接口屏蔽底层模型差异。
#### C. RAG 知识引擎 (Retriever)
- **ETL 流程**：
    - **读取**：支持 PDF, Word, Markdown 格式。
    - **切片 (Chunking)**：按段落 + 语义切分。
    - **写入**：存入 **PGvector** (架构图中的 VectorStore)。
- **检索增强**：采用混合检索（关键词 + 向量），并基于元数据（`city=qingdao`）进行 Filter 过滤。
#### D. 工具与 MCP (Tools & MCP)
- **本地工具 (Tools)**：
    - **补贴计算器**：Java 编写的本地 Function，用于精确计算数字。
    - **文件处理**：解析用户上传的发票或旧机参数表。
- **MCP 集成 (Model Context Protocol)**：
    - 使用 MCP 协议连接**联网搜索服务**（抓取最新的政府公告）。
    - 使用 MCP 连接**外部 API**（如京东/苏宁的旧机回收估价接口）。

### 3.3 数据层 (External Dependencies)
- **向量数据库**：PostgreSQL + PGvector 插件（存储高维向量）。    
- **关系型数据库**：存储结构化政策表（如：补贴比例字典表）、用户日志。
- **文件存储**：对象存储（OSS/MinIO），用于存原始 PDF 文件。

### 3.4 后台管理系统 (Admin Web)
- **RAG 管理**：可视化的 ETL 任务监控，查看切片效果，手动修正向量库中的错误数据。
- **Advisors 配置**：动态配置敏感词列表，开关 ReReading 机制。
- **日志分析**：基于“日志 Advisor”产出的数据，展示 Token 消耗趋势、用户提问热力图。

## 4. 技术架构选型 (Technical Architecture)
基于提供的架构图，技术栈更加明确：
- **基础设施**：
    - **部署方式**：Docker 容器化部署 / Serverless（如阿里云 FC，对应架构图）。
    - **Web 容器**：Tomcat / Undertow。
- **后端框架**：
    - **语言**：Java 21 (LTS).
    - **核心框架**：Spring Boot 3.x.
    - **AI 框架**：**Spring AI** (核心依赖，提供 ChatClient, Advisors, RAG, MCP 支持).
- **AI 模型层**：
    - **推理模型**：DeepSeek-V3 / R1 (通过 API 接入), 通义千问 (备选).
    - **Embedding**：BGE-M3 或 OpenAI text-embedding-3.
- **数据存储**：
    - **Vector DB**：PostgreSQL (PGvector).
    - **Chat Memory**：Redis (生产环境) 或 In-Memory (开发环境).

## 5. 项目实施阶段规划 (Roadmap)

### 第一阶段：MVP 原型 (3周)
- **重点**：跑通 **RAG + ChatClient** 流程。
- **内容**：
    - Spring AI 基础环境搭建。
    - 实现 PDF 的 ETL 入库（写入 VectorStore）。
    - 搭建微信小程序，实现基础问答。
    - **无 Advisors，无 MCP**。

### 第二阶段：Agent 增强 (4周)
- **重点**：引入 **Advisors** 和 **Tools**。
- **内容**：
    - 加入 **ChatMemory Advisor** 实现多轮对话。
    - 加入 **ReReading Advisor** 提升准确率。
    - 开发“补贴计算器” Tool，并通过 Function Calling 调用。
    - 接入 DeepSeek-R1 等强推理模型。
### 第三阶段：MCP 与 生产化 (3周)
- **重点**：**MCP 协议**落地与系统稳健性。
- **内容**：
    - 将联网搜索封装为 MCP Server。
    - 上线后台管理系统，监控日志。
    - Docker 镜像构建与 ECS/Serverless 部署。
### 第四阶段：上线运维
- **内容**：生产环境压力测试，Token 成本优化。
